Phase 14: CAG Suite Check Enhancements
===========================================
Proposed additions to the CAG check suite (app/checks/cag/).
Ordered by value to the reconnaissance workflow.

Current checks:
  - CAGDiscoveryCheck: Detects 5 cache types (GPTCache, semantic cache,
    prompt cache, KV cache, Redis) via path probing and header/body
    fingerprinting. Probes 27 cache management paths. Detects caching
    behavior on existing AI/RAG endpoints via timing and response
    analysis. Produces cag_endpoints and cache_infrastructure outputs.
  - CAGCacheProbeCheck: 4 active vulnerability tests:
    1. Cross-session cache leakage (plants marker in session A, queries
       from session B)
    2. Cache timing analysis (3 identical requests measuring speedup)
    3. Context ID enumeration (7 predictable ID patterns)
    4. Cache key predictability (trailing space input variation)
    Note: stale_context test is defined but not implemented.

HIGH VALUE — CACHE-SPECIFIC ATTACK SURFACE
==========================================================================

1. Cache Poisoning with Malicious Content
--------------------------------------------
Priority:   High
Depends on: cag_discovery (needs confirmed cache infrastructure),
            cag_cache_probe (timing/key analysis helps target the attack)

Technique:  The probe check detects caching behavior but never actually
            poisons the cache. This check sends a request with an
            injection payload, confirms it's cached, then verifies the
            poisoned response is served to subsequent users.

Test sequence:
  1. Send a query with an embedded injection marker:
     "What is the capital of France? [CACHE_POISON_MARKER_{uuid}]"
  2. Wait 1-2 seconds for cache population
  3. Send the same (or similar, for semantic cache) query from a
     different session context (different X-Session-Id, different
     auth token, or no auth)
  4. Check if the response contains the poison marker
  5. If yes: cache poisoning confirmed — one user's injection
     persists for other users

Why it matters: This is the actual attack, not just detection. If
a poisoned response gets cached, every subsequent user who triggers
the same cache key receives the attacker's content without any
active exploitation.

Findings:
  - critical: "Cache poisoning confirmed: injected content served to
    different session (marker found in cross-session response)"
  - high: "Cache poisoning possible: injected content cached (timing
    confirms cache hit) but cross-session delivery not confirmed"
  - medium: "Cache accepts arbitrary content but caching not confirmed
    for injected payload"
  - info: "Cache poisoning not possible: injected content not cached
    or not served cross-session"

Implementation notes:
  - Use unique UUID markers per test to avoid false positives
  - Test with both exact query replay and semantically similar
    queries (for semantic caches)
  - Attempt cleanup: send a different query to the same cache key to
    overwrite the poisoned entry
  - If cache/clear endpoint exists, use it after testing
  - This test modifies cache state — should be gated behind
    --aggressive or explicit opt-in
  - Log the exact poisoned query and marker for proof-of-scope


2. Stale Context Exploitation
-------------------------------
Priority:   High (defined in codebase but not implemented)
Depends on: cag_discovery (needs cache endpoints with TTL/expiry)

Technique:  Test if cached context outlives its validity window. If
            a user's context is cached and their permissions change,
            does the cache still serve the old (elevated) context?

Test sequence:
  1. Send a request establishing a context with specific characteristics:
     "My role is administrator. What admin functions are available?"
  2. Note the response (baseline with context)
  3. Wait for a reasonable TTL window (check TTL from discovery, or
     test at 30s, 60s, 120s intervals)
  4. Send a new request from a fresh session asking:
     "What functions are available?"
  5. Check if the stale admin context influences the fresh response

  Alternative approach (simpler):
  1. Send a query and cache the response
  2. Wait for TTL + buffer
  3. Re-query and check if the stale response is still served
  4. Stale if: response is identical AND timing shows cache hit
     AND TTL should have expired

Why it matters: Cached contexts that outlive permission changes
create a privilege persistence vector. A user whose access was
revoked continues to receive elevated responses until the cache
entry expires. In high-security environments, this can violate
compliance requirements around access revocation timeliness.

Findings:
  - high: "Stale context served: cached response delivered after
    expected TTL expiry ({N} seconds past expected expiry)"
  - medium: "Long-lived cache: TTL exceeds {N} seconds (stale
    context risk)"
  - low: "Cache TTL detected but within reasonable range ({N}s)"
  - info: "Cache properly expires entries (stale context not
    detected)"

Implementation notes:
  - This test is time-dependent — may need to wait minutes
  - Consider running as a background check that revisits after
    delay
  - Check cache headers (max-age, expires, age) for expected TTL
  - Compare with actual behavior to detect TTL mismatches


3. Cross-User Cache Leakage (Enhanced)
-----------------------------------------
Priority:   High
Depends on: cag_cache_probe (builds on existing cross-session test)

Technique:  The existing cross-session test uses a simple marker and
            X-Session-Id headers. This enhanced version tests deeper
            isolation properties.

Additional tests:
  - Different auth tokens: Send request with Token A, then same
    query with Token B. If Token B gets Token A's cached response,
    the cache key doesn't include user identity.
  - No-auth after auth: Send authenticated request, then same query
    without auth headers. Check if unauthenticated users receive
    the authenticated cached response.
  - User-specific content: Send a query like "Show my account
    details" with User A context, then same query with User B
    context. If responses match, per-user cache isolation is broken.
  - API key variation: Different API keys or Bearer tokens with
    same query to test if auth is part of cache key.

Why it matters: The basic marker test might miss cases where the
cache is session-aware but not user-aware, or where auth tokens
aren't included in the cache key but session IDs are.

Findings:
  - critical: "Cross-user cache leakage: authenticated response
    served to unauthenticated request (cache key excludes auth)"
  - critical: "Cross-user cache leakage: User A's response served
    to User B (cache key excludes user identity)"
  - high: "Cache key does not include API key: different tokens
    receive same cached response"
  - medium: "Cache key includes session ID but not user identity
    (session fixation enables cross-user leakage)"
  - info: "Cache properly isolates responses per user"

Implementation notes:
  - Requires at least two different auth contexts to test
  - If only one auth context available, test auth vs no-auth
  - The no-auth-after-auth test is the simplest and most common
    vulnerability


4. Cache-Based Prompt Injection Persistence
----------------------------------------------
Priority:   High
Depends on: cache_poisoning (check #1), indirect_injection results
            from RAG suite

Technique:  Combine cache poisoning with prompt injection to create
            persistent, multi-user compromise. If an injection payload
            gets cached, every subsequent user who triggers that cache
            key receives the injected response.

Test sequence:
  1. Identify a query that triggers prompt injection (from RAG
     indirect injection or AI prompt leakage results)
  2. Send the injection query and confirm the response contains
     injected content
  3. Verify the response is cached (timing analysis or cache headers)
  4. Send the same query from a different user context
  5. Check if the new user receives the injected response

Escalation:
  - If semantic cache: test if similar (but not identical) queries
    also serve the poisoned response, expanding the blast radius
  - Map the "injection radius" — how many query variations serve
    the poisoned response?

Findings:
  - critical: "Persistent injection via cache: prompt injection
    response cached and served to other users"
  - critical: "Semantic cache amplifies injection: poisoned response
    served to {N} query variations"
  - high: "Injection response cached but cross-user delivery
    unconfirmed"
  - info: "Injection responses not cached"

Implementation notes:
  - Requires successful injection from another check as input
  - This is the highest-impact CAG finding — single injection
    affects all users until cache expires
  - Must attempt cleanup after testing
  - Log exact injection query and cache key for remediation


5. Cache Eviction/Invalidation Testing
-----------------------------------------
Priority:   High
Depends on: cag_discovery (needs cache management endpoints)

Technique:  Test if cache management endpoints (clear, flush,
            invalidate) are accessible and what happens when the cache
            is forcibly evicted.

Tests:
  - Probe without auth:
    POST /cache/clear
    POST /cache/flush
    POST /cache/invalidate
    DELETE /cache
    POST /cache/purge
  - Check for key-specific eviction:
    DELETE /cache/{key}
    POST /cache/invalidate?key={key}
  - After eviction: verify cache is actually cleared (timing analysis
    should show cache miss on previously cached queries)

Why it matters:
  - DoS via cache busting: attacker clears cache, forcing expensive
    recomputation for all users
  - Poisoning preparation: clear cache, then immediately fill it
    with poisoned entries before legitimate traffic repopulates
  - Availability impact: if cache is critical to performance, clearing
    it may cause cascading failures

Findings:
  - critical: "Cache clear endpoint accessible without auth:
    POST /cache/clear returned 200"
  - high: "Cache eviction possible: specific cache keys deletable
    via DELETE /cache/{key}"
  - medium: "Cache management endpoint exists but requires auth:
    POST /cache/clear returned 401"
  - info: "No cache management endpoints accessible"

Implementation notes:
  - Test eviction BEFORE poisoning checks (knowing eviction is
    possible changes the risk profile)
  - Be careful with /cache/clear on production — it could impact
    real users. Prefer key-specific eviction testing.
  - Gate behind --aggressive for clear/flush endpoints


6. Semantic Cache Similarity Threshold Probing
-------------------------------------------------
Priority:   High (if semantic cache detected)
Depends on: cag_discovery (needs confirmed semantic cache / GPTCache)

Technique:  Semantic caches use embedding similarity to determine
            cache hits. Probe the similarity threshold to understand
            how "different" a query needs to be to get a fresh response
            vs a cached one.

Tests:
  1. Establish baseline: Send query A, confirm it's cached (timing)
  2. Send variations with decreasing similarity:
     - Exact duplicate: "What is the capital of France?"
     - Minor variation: "What's the capital of France?"
     - Rephrased: "Tell me France's capital city"
     - Related: "What is the main city of France?"
     - Tangential: "Tell me about France"
     - Unrelated: "What is the weather today?"
  3. For each variation: measure response time and compare response
     content to baseline
  4. Map the threshold: at what point does the cache stop matching?

Why it matters: A loose threshold (0.7 similarity) means a poisoned
cache entry for "company financial data" might also serve queries
for "company revenue," "business finances," and "corporate earnings."
The looser the threshold, the wider the blast radius of any cache
poisoning attack.

Findings:
  - high: "Loose semantic cache threshold: poisoned entry could serve
    {N} query variations (estimated threshold: {similarity})"
  - medium: "Semantic cache hit on rephrased queries (threshold
    estimated at {similarity})"
  - low: "Semantic cache only matches near-exact queries (tight
    threshold)"
  - info: "Not a semantic cache (exact match only)"

Implementation notes:
  - Requires multiple requests — rate limit carefully
  - Use timing as the primary signal (cache hit = fast)
  - The similarity threshold estimate is approximate — based on
    which variations hit vs miss
  - Report the threshold estimate to help operators configure it

7. Cache Key Reverse Engineering
-----------------------------------
Priority:   Medium
Depends on: cag_cache_probe (builds on key predictability test)

Technique:  Systematically vary query components to map exactly which
            parts of the input determine the cache key.

Variations to test:
  - Capitalization: "capital of france" vs "Capital of France"
  - Punctuation: "capital of france" vs "capital of france?"
  - Word order: "france capital what is" vs "what is france capital"
  - Synonyms: "capital" vs "main city"
  - Prefix/suffix: "Please tell me the capital" vs "capital"
  - Whitespace: extra spaces, tabs, newlines
  - System prompt variation: same query with different system message
  - Model parameter variation: same query with different temperature

For each pair: send first variation, wait for cache, send second
variation, check if cache hit occurs.

Why it matters: If the cache key only considers the first N tokens,
an attacker can append injection payload after the key boundary. If
the key ignores system prompts, different users sharing the same
cache despite different safety configurations.

Findings:
  - high: "Cache key excludes system prompt: different system
    messages share cache entries"
  - high: "Cache key uses only first {N} tokens: content after
    boundary can be injected without changing cache key"
  - medium: "Cache key is case-insensitive: capitalization variations
    hit same cache entry"
  - low: "Cache key includes most query components (detailed key)"
  - info: "Cache key could not be mapped (insufficient data)"

Implementation notes:
  - This generates many requests — rate limit and consider making
    it opt-in
  - Binary search on token boundaries to find key cutoff
  - The system prompt exclusion test is especially valuable for
    multi-tenant systems


8. TTL/Expiry Mapping
-----------------------
Priority:   Medium
Depends on: cag_discovery (needs confirmed caching)

Technique:  Determine how long cached entries live by re-querying at
            increasing intervals.

Tests:
  - Send unique query, confirm cached (timing/headers)
  - Re-query at: 5s, 15s, 30s, 60s, 120s, 300s intervals
  - Detect transition from cache hit to cache miss
  - Check cache-related headers: max-age, Age, Expires, Cache-Control
  - Compare header-stated TTL with actual observed TTL

Findings:
  - medium: "Cache TTL: {N} seconds (long-lived cache increases
    poisoning exposure window)"
  - low: "Cache TTL header present: max-age={N}"
  - low: "Cache TTL mismatch: header says {N}s but cache persisted
    for {M}s"
  - info: "Cache TTL: {N} seconds (reasonable)"

Implementation notes:
  - This check is inherently slow (may need to wait minutes)
  - Consider running as a background check
  - Short TTL (< 60s) = lower risk; long TTL (> 300s) = higher risk
  - No TTL at all = indefinite caching = highest risk


9. Cache Warming Abuse
------------------------
Priority:   Medium
Depends on: cag_discovery (needs cache management endpoints)

Technique:  If /cache/warm or /precompute endpoints are accessible,
            test if an attacker can pre-populate the cache with
            chosen content.

Tests:
  - POST /cache/warm with crafted content:
    {"query": "sensitive topic", "response": "attacker controlled content"}
  - POST /precompute with arbitrary prompts
  - POST /cache/store with key-value pairs
  - After warming: query the topic and check if pre-populated
    content is served

Findings:
  - critical: "Cache warming accepts arbitrary content: attacker can
    pre-populate cache via /cache/warm"
  - high: "Precompute endpoint accepts arbitrary prompts:
    /precompute accessible without auth"
  - medium: "Cache warming endpoint exists but validates input"
  - info: "No cache warming endpoints accessible"

Implementation notes:
  - Only test if discovery found /cache/warm, /precompute, or
    /cache/store endpoints
  - This is a write operation — gate behind --aggressive
  - Attempt cleanup after testing


10. Multi-Layer Cache Detection
---------------------------------
Priority:   Medium
Depends on: cag_discovery (needs confirmed caching)

Technique:  Production systems often have multiple cache layers
            (CDN -> application cache -> semantic cache -> KV cache).
            Test whether bypassing one layer reveals another.

Tests:
  - Send request normally (hits all cache layers)
  - Send with Cache-Control: no-cache (bypass HTTP cache layer)
  - Send with Pragma: no-cache (bypass older HTTP caches)
  - Send with unique cache-buster parameter (?_cb=timestamp)
  - Compare timing and response content across variations
  - If timing differs: different layers have different latency
    profiles

Findings:
  - medium: "Multiple cache layers detected: {N} layers with
    different bypass behavior"
  - low: "HTTP cache layer bypassable: Cache-Control: no-cache
    reaches origin but semantic cache still active"
  - info: "Single cache layer detected"
  - info: "Cache-busting headers respected"

Implementation notes:
  - The key insight is that different layers respond differently
    to bypass headers
  - CDN layer: bypassed by Cache-Control headers
  - Application cache: may ignore HTTP cache headers
  - Semantic cache: ignores all HTTP cache semantics
  - Each layer may have different security properties


11. Cache Side-Channel: Content Inference
--------------------------------------------
Priority:   Medium
Depends on: cag_cache_probe (needs timing analysis capability)

Technique:  Use timing differences to infer what other users have
            queried without seeing their queries or responses. A cache
            hit (fast response) for topic X means someone recently
            queried topic X.

Tests:
  - Send queries about sensitive topics and measure response time:
    "salary information", "merger plans", "employee performance",
    "api keys", "security vulnerabilities"
  - Cache hit (fast) = topic recently queried by another user
  - Cache miss (slow) = topic not recently queried
  - Build a profile of which topics are "hot" in the cache

Why it matters: Even without seeing cached content, the timing
oracle reveals user behavior patterns. If "salary data" consistently
shows cache hits, someone is regularly querying salary information.
This is a privacy leak via side channel.

Findings:
  - medium: "Cache timing side-channel: {N}/{M} sensitive topic
    queries showed cache hits (recent user activity inferred)"
  - low: "Cache timing oracle available: response time varies
    significantly between cached and uncached queries"
  - info: "Timing analysis inconclusive (no clear cache hit/miss
    pattern)"

Implementation notes:
  - Requires multiple measurements per topic for reliability
  - Network jitter can cause false positives — use statistical
    analysis (mean + stddev over 3-5 measurements)
  - This is passive reconnaissance — doesn't modify cache state
  - Ethical concern: this reveals other users' query patterns.
    Report the capability, not the specific topics.


12. Cache Quota/Size Exploitation
------------------------------------
Priority:   Medium
Depends on: cag_discovery (needs confirmed caching)

Technique:  Test if the cache has size limits and what happens when
            they're reached.

Tests:
  - Send N unique queries (50-100) with unique content to fill cache
  - After filling: re-query the first few entries. If they're
    evicted (cache miss), the cache uses LRU or similar eviction.
  - Check if the cache stops accepting new entries (fixed size)
  - Measure the approximate cache size based on eviction point
  - Test if an attacker can flush valuable entries by flooding with
    garbage

Findings:
  - medium: "Cache exhaustion possible: {N} unique queries evicted
    existing entries (LRU eviction confirmed)"
  - medium: "Unbounded cache: {N} queries accepted with no eviction
    (memory exhaustion risk)"
  - low: "Cache size limited: approximately {N} entries before
    eviction"
  - info: "Cache quota testing inconclusive"

Implementation notes:
  - Use small, unique queries to minimize token cost
  - This generates many requests — respect rate limits
  - Don't actually exhaust the cache on production — estimate from
    the eviction threshold
  - Gate behind --aggressive

13. Provider-Specific Prompt Caching Analysis
-------------------------------------------------
Priority:   Low-medium
Depends on: cag_discovery (needs AI endpoints with usage metadata)

Technique:  OpenAI and Anthropic offer server-side prompt caching
            (reported via cached_tokens in usage metadata). Test if
            the application properly handles cached vs non-cached
            responses and if caching leaks information.

Tests:
  - Send a request and check usage.cached_tokens in response
  - Compare cached_tokens across different queries:
    - Same system prompt + different user message: high cached tokens
      = shared system prompt prefix
    - Different system prompt: cached tokens should drop
  - If cached_tokens is unexpectedly high for a unique query, the
    cache includes another user's prefix (cross-user system prompt
    sharing)

Findings:
  - medium: "Provider caching reveals shared system prompt: {N}
    cached tokens suggest prefix sharing across users"
  - low: "Provider prompt caching active: {N} cached tokens reported"
  - info: "No provider-level prompt caching detected"

Implementation notes:
  - Only relevant for OpenAI and Anthropic-backed services
  - Check for usage.cached_tokens, usage.prompt_tokens_cached fields
  - This is informational — the caching is server-side and can't
    be poisoned by the client


14. Cache Serialization Attacks
---------------------------------
Priority:   Low
Depends on: vector_store_direct_access or redis_cache_access

Technique:  If the cache stores serialized objects, test if
            deserialization of a crafted cache entry can trigger
            code execution.

Tests:
  - Identify cache backend (Redis, filesystem, memcached)
  - If direct cache access: check serialization format (pickle,
    JSON, msgpack, protobuf)
  - If pickle: this is a critical vulnerability — pickle
    deserialization enables arbitrary code execution
  - If JSON with type hints: test for type confusion
  - If filesystem: check for path traversal in cache key → filename
    mapping

Findings:
  - critical: "Cache uses pickle serialization with writable access:
    arbitrary code execution via deserialization"
  - high: "Cache uses unsafe deserialization: {format} with writable
    backend"
  - medium: "Cache backend directly accessible: {backend} at {url}"
  - info: "Cache serialization appears safe (JSON without type hints)"

Implementation notes:
  - Requires direct cache backend access (not just the application
    layer)
  - Pickle deserialization is an immediate critical finding — no
    need to demonstrate RCE
  - This is more of a traditional infrastructure finding than
    CAG-specific
  - Redis without auth + pickle = critical chain


15. Distributed Cache Consistency
------------------------------------
Priority:   Low
Depends on: cag_discovery, multi-node infrastructure

Technique:  If multiple cache nodes exist (Redis cluster, memcached
            ring), test if writes to one node propagate to all.

Tests:
  - Identify cache topology from headers (x-cache-node, x-served-by)
  - Send same query multiple times and check which node responds
    (from headers)
  - Poison cache on one node, check if other nodes serve the poison
  - Check replication lag by timing propagation

Findings:
  - medium: "Cache replication inconsistency: poisoned entry on
    node A not visible on node B (partial poisoning possible)"
  - low: "Multiple cache nodes detected: {N} nodes identified from
    response headers"
  - info: "Single cache node or consistent replication"

Implementation notes:
  - Hard to test without infrastructure knowledge
  - Look for x-cache-node, x-served-by headers that vary
  - Inconsistent replication means poisoning is harder to detect
    (only some users affected)
