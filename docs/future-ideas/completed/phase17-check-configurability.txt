Phase 17: Check Configurability
==========================================
Make check behavior tunable without touching Python. Currently, check-
specific parameters (wordlists, endpoint paths, payloads, timeouts,
rate limits) are hardcoded as class constants or constructor defaults.
The config system (chainsmith.yaml) only handles global settings.

This phase introduces per-check configuration, check profiles, and
externalized payload data so operators can adapt checks to each
engagement without code changes.


Current state:
  - Global config: chainsmith.yaml covers scope, ports, LiteLLM,
    storage, swarm — no per-check knobs
  - Check parameters: Hardcoded in class bodies (e.g.,
    DnsEnumerationCheck.DEFAULT_WORDLIST, LLMEndpointCheck.CHAT_PATHS,
    APIParameterInjectionCheck.PARAM_TESTS)
  - Filtering: ScanStartInput accepts checks/suites lists at scan-start;
    no persistent enable/disable per check
  - Behavior tuning: timeout_seconds, requests_per_second,
    delay_between_targets, retry_count are BaseCheck attributes set at
    instantiation, not overridable from config
  - on_critical: Suite-level only, not per-check

WAVE 1 — PER-CHECK CONFIG SECTION IN YAML
==========================================================================

Add a `checks:` section to chainsmith.yaml that layers over class
defaults. Every BaseCheck attribute becomes overridable per check.

Schema:
  checks:
    _defaults:                        # Applies to all checks
      timeout_seconds: 30
      requests_per_second: 10
      retry_count: 1
    dns_enumeration:
      wordlist: ["api", "chat", "admin", "internal", "staging"]
      timeout_seconds: 60
    llm_endpoint_discovery:
      chat_paths: ["/v1/chat/completions", "/custom/chat"]
      embedding_paths: ["/v1/embeddings"]
      requests_per_second: 5
    header_analysis:
      critical_headers: ["Content-Security-Policy", "X-Frame-Options"]
    api_parameter_injection:
      enabled: false                  # Skip entirely

Resolution order (last wins):
  1. Hardcoded class default
  2. checks._defaults
  3. checks.<check_name>
  4. Constructor kwarg (for programmatic use)
  5. Environment variable CHAINSMITH_CHECK_<NAME>_<PARAM>

Implementation notes:
  - Modify BaseCheck.__init__ to pull overrides from get_config()
  - Config is read once at check instantiation, not per-execution
  - Unknown keys under a check name are passed to the check's __init__
    as kwargs — checks opt in to custom params via constructor
  - Validate early: fail at load time if a check name in config doesn't
    match any registered check (typo protection)

WAVE 2 — EXTERNALIZE PAYLOAD DATA
==========================================================================

Extract test cases, wordlists, and endpoint path lists from Python
into data files. Checks become engines that load their payloads at
init from a configurable path.

Proposed layout:
  data/
    payloads/
      jailbreak_prompts.yaml
      parameter_injection.yaml
      mcp_tool_injection.yaml
      prompt_leakage_probes.yaml
      input_format_injection.yaml
    wordlists/
      subdomains_small.txt          # 16 entries (current default)
      subdomains_full.txt           # Expanded list
      paths_web.txt
      paths_ai.txt
      model_names.txt
    endpoints/
      chat_paths.yaml               # LLMEndpointCheck.CHAT_PATHS
      embedding_paths.yaml           # EmbeddingEndpointCheck paths
      model_info_paths.yaml          # ModelInfoCheck.MODEL_PATHS

Config integration:
  checks:
    dns_enumeration:
      wordlist_file: "./data/wordlists/subdomains_full.txt"
    jailbreak_testing:
      payload_file: "./data/payloads/jailbreak_prompts.yaml"
      depth: standard                # light | standard | thorough

Benefits:
  - Non-developers can update payloads without touching Python
  - Payloads can be swapped per-engagement via config
  - Community payload contributions are easy
  - Version-controlled payload evolution (git diff on data files)

Implementation notes:
  - Add a load_payload(path, default) utility in app/lib/
  - Relative paths resolve from project root
  - Absolute paths honored as-is
  - Each check retains its current hardcoded values as fallback if
    no data file is specified
  - YAML for structured payloads (test name, value, severity, etc.);
    plain text for simple wordlists (one entry per line)

WAVE 3 — CHECK PRESETS
==========================================================================

Extend the existing port_profile concept to full check presets.
A preset bundles suite selection, check enablement, and parameter
overrides into a single switchable configuration.

Schema:
  presets:
    quick:
      suites: [network, web]
      checks:
        dns_enumeration:
          wordlist_file: "./data/wordlists/subdomains_small.txt"
        _defaults:
          timeout_seconds: 15

    thorough:
      suites: [network, web, ai, mcp, agent, rag, cag]
      checks:
        dns_enumeration:
          wordlist_file: "./data/wordlists/subdomains_full.txt"
        _defaults:
          timeout_seconds: 120
          retry_count: 3

    passive:
      exclude_intrusive: true
      checks:
        _defaults:
          requests_per_second: 2
          delay_between_targets: 1.0

    ai-focused:
      suites: [network, ai, mcp, agent, rag, cag]
      checks:
        port_scan:
          port_profile: ai
        _defaults:
          timeout_seconds: 90

Usage:
  - CLI: chainsmith scan --preset thorough
  - API: ScanStartInput gains a preset field
  - Config: default_preset: quick in chainsmith.yaml

Resolution order:
  1. Preset values
  2. Top-level checks: values (preset-independent overrides)
  3. Runtime overrides (CLI flags, API params)

Implementation notes:
  - Presets are pure config — no code changes to checks themselves
  - Preset merges with top-level config; explicit values win
  - Preset "inherits" from base config, not from other presets
    (no preset chaining — keep it simple)

WAVE 4 — PER-CHECK ENABLE/DISABLE AND ON_CRITICAL
==========================================================================

4a. Persistent check enablement
---------------------------------
Add enabled flag per check in config, checked in can_run() before
condition evaluation.

  checks:
    api_parameter_injection:
      enabled: false
      reason: "Client prohibits parameter fuzzing"  # Optional, for docs

This is simpler than passing exclusion lists through the API every
scan. The API checks/suites filters still work — this is the persistent
layer underneath.

Implementation:
  - BaseCheck gains self.enabled (default True)
  - can_run() returns False if not enabled
  - CheckLauncher logs skipped-because-disabled separately from
    skipped-because-conditions-not-met


4b. Per-check on_critical
---------------------------
Currently on_critical is suite-level. Some checks within a suite
warrant different responses.

  checks:
    port_scan:
      on_critical: annotate
    tls_analysis:
      on_critical: skip_downstream

Resolution: per-check > per-suite > default.

Implementation:
  - BaseCheck gains self.on_critical (default None = inherit from suite)
  - CheckLauncher checks check-level first, then falls back to suite


==========================================================================
WAVE 5 — ENVIRONMENT VARIABLE OVERRIDES AND VALIDATION
==========================================================================

5a. Env var overrides for check params
----------------------------------------
Extend the existing CHAINSMITH_* pattern to individual check params:

  CHAINSMITH_CHECK_DNS_ENUMERATION_TIMEOUT_SECONDS=60
  CHAINSMITH_CHECK_LLM_ENDPOINT_DISCOVERY_REQUESTS_PER_SECOND=2
  CHAINSMITH_CHECK_API_PARAMETER_INJECTION_ENABLED=false

Convention:
  CHAINSMITH_CHECK_<CHECK_NAME>_<PARAM> (all uppercase, hyphens to
  underscores)

Essential for CI/CD and containerized deployments where mounting
config files is inconvenient.

Implementation:
  - In BaseCheck.__init__, after YAML config lookup, scan os.environ
    for matching prefix
  - Type coercion: bool for enabled/intrusive/sequential, float for
    rates/delays, int for timeouts/retries, str for paths
  - Env vars win over YAML (highest priority after constructor kwargs)


5b. Config validation
-----------------------
Add validate_config() classmethod to BaseCheck. Called at load time
so misconfigs fail fast.

Validations:
  - timeout_seconds > 0
  - requests_per_second > 0
  - retry_count >= 0
  - wordlist/payload files exist if specified
  - Check names in config match registered checks (typo protection)
  - on_critical is one of: annotate, skip_downstream, stop
  - Preset references valid suites

On failure: raise ConfigError with clear message including the check
name, the bad value, and what was expected.

WAVE 6 — WEBUI CHECK DETAIL MODALS
==========================================================================

Add a modal to each check in the index.html grid that serves two
purposes: (1) explain what the check does, and (2) allow per-check
configuration changes before scan invocation.

Interaction model:
  - Checkbox click: toggles selection (unchanged from current behavior)
  - Label / card click: opens the check detail modal
  - An info icon on each check item hints that details are available
  - Modal contains a toggle to enable/disable the check, so users can
    also control selection from within the modal

Modal content — info section:
  - Check name (human-readable, underscores → spaces)
  - Description (from check metadata)
  - Why run this (reason field)
  - References (OWASP, CWE, RFC links — rendered as clickable list)
  - Techniques (MITRE ATT&CK / methodology tags — rendered as pills)
  - Conditions (what context must exist for this check to run)
  - Produces (what context data this check generates)
  - Intrusive badge (if applicable)

Modal content — configuration section (Wave 1 dependency):
  - Enable/disable toggle (synced with the grid checkbox)
  - Per-check parameter fields, dynamically rendered from the check's
    configurable params. Each check declares its tunable parameters
    (Wave 1); the modal renders appropriate form controls:
      - Text input for strings (wordlist paths, endpoint paths)
      - Number input for numeric values (timeout, rate limit, retry)
      - Select/dropdown for enum values (on_critical, depth level)
      - Toggle for booleans (enabled, intrusive override)
      - Textarea for list values (comma-separated or one-per-line)
  - on_critical override (per-check, overrides suite default)
  - A "Reset to defaults" link that clears per-check overrides

Locked intrusive checks:
  - Modal still opens (users should be able to learn what a check does
    even when it's locked)
  - Enable toggle is disabled with explanatory text:
    "This is an intrusive check. Enable Active Testing globally, then
    unlock this suite's intrusive checks to configure it."
  - Parameter fields are visible but disabled (read-only preview)
  - Visual treatment: muted header bar or banner at top of modal
    indicating locked state

Data flow:
  - Modal reads check metadata from the already-fetched /api/v1/checks
    response (no additional API call for info)
  - Parameter definitions need a new field in check metadata — either
    a configurable_params list from the check class, or derived from
    the YAML schema (Wave 1)
  - On modal close, changed parameters are held in a client-side
    config overlay object (not persisted until scan start)
  - At scan submission, the overlay merges into the scope payload
    alongside the existing scan_behavior and techniques fields
  - API receives per-check overrides in a new check_config field:
      {
        "check_config": {
          "dns_enumeration": {
            "timeout_seconds": 60,
            "wordlist_file": "./data/wordlists/subdomains_full.txt"
          },
          "port_scan": {
            "on_critical": "skip_downstream"
          }
        }
      }

Implementation notes:
  - Store full check metadata objects in CHECK_SUITES (not just names)
    so the modal can render without re-fetching
  - Reuse existing modal infrastructure from common.css (.modal,
    .modal-overlay, .modal-header, .modal-body, .modal-section, etc.)
  - Single shared modal element in the DOM, populated dynamically on
    open (not one modal per check)
  - Prevent label click from bubbling to checkbox (stopPropagation on
    the label handler, or restructure the HTML so label and checkbox
    are siblings rather than parent-child)
  - CSS: info icon uses a subtle SVG or Unicode character (e.g., a
    small circle-i) positioned at the right edge of the check item
  - Transition: if Wave 1 is not yet implemented, the config section
    shows only enable/disable and on_critical. Parameter fields appear
    once checks declare their configurable params.

Depends on:
  - Wave 1 (per-check config in YAML) for parameter field rendering
  - Wave 4 (per-check on_critical) for the on_critical dropdown
  - Partial standalone value: info-only modals with enable/disable
    can ship before Waves 1 and 4 land

OPEN QUESTIONS
==========================================================================

1. Should presets support inheritance (preset extends another
   preset)? Adds complexity but reduces duplication. Current
   recommendation: no, keep it flat.

2. Should payload files support Jinja2 templating for target-specific
   payload generation? (e.g., injecting the target domain into
   payloads.) Or is that over-engineering at this stage?

3. How should the config handle check names that don't exist yet
   (e.g., from a not-yet-installed plugin)? Strict validation would
   reject them. Lenient mode would warn and skip. Recommendation:
   strict by default, with a config flag to allow unknown check names.

4. Should there be a config export command that dumps the fully
   resolved config (all layers merged) for debugging? Useful for
   CI/CD troubleshooting.

5. Should per-check config support conditional values based on
   target properties? (e.g., "use this wordlist if target is .gov,
   that wordlist otherwise.") Recommendation: not in this phase —
   use profiles to handle different engagement types.

6. Where should the data/ directory live relative to the project?
   Options: project root (current recommendation), user home
   (~/.chainsmith/data/), or configurable via data_dir setting.

7. Should enabled: false checks still appear in /api/checks output
   (marked as disabled) or be hidden entirely? (Wave 6 assumes yes —
   disabled checks are visible in the grid and openable in the modal.)

8. "Profile" is already used by chainsmith.sh for AI provider
   selection. This doc uses "preset" to avoid collision.
   The port_profile field keeps its name since it predates this work.


==========================================================================
DESIGN DECISIONS TO RESOLVE
==========================================================================

Wave 1 — Per-check config
----------------------------
D1. Check name format in YAML: The doc uses snake_case
    (dns_enumeration). Should this match the Python class name, a slug
    derived from it, or the name attribute on the check? Need a
    consistent mapping rule.

D2. Unknown kwargs: The doc says unknown keys get passed as kwargs to
    the check constructor. That's flexible but error-prone — a typo
    silently becomes a kwarg. Should checks be required to declare
    their accepted custom params explicitly?

Wave 2 — Externalized payloads
---------------------------------
D3. Depth levels (light | standard | thorough): Is this a per-file
    concept (three separate files per payload type) or a single file
    with entries tagged by depth? Tagged entries in one file are easier
    to maintain but more complex to parse.

D4. Data dir location: Project root as default with a data_dir
    setting in chainsmith.yaml? Or user home (~/.chainsmith/data/)?
    Or always relative to project root?

Wave 3 — Presets
-------------------
D5. Where do presets live? Inline in chainsmith.yaml, or separate
    files under presets/quick.yaml, presets/thorough.yaml? Separate
    files are cleaner for sharing/versioning but add file sprawl.

D6. Default preset: Should there always be a default preset, or is
    "no preset" (just raw config) the default behavior?

Wave 4 — Enable/disable + on_critical
-----------------------------------------
D7. Disabled checks in API/UI: Should disabled checks appear in
    /api/checks output (marked as disabled) so the WebUI can offer a
    toggle to re-enable? Or hidden entirely?

D8. The "reason" field on enabled: false — does anything consume it
    (shown in UI, included in scan reports), or is it purely a
    config-file comment for human readers?

Wave 5 — Env vars + validation
----------------------------------
D9. Strict vs lenient for unknown check names in WebUI context: If
    someone saves a config with a check name that doesn't load until
    a plugin is installed, should we block the save, warn, or silently
    accept?

Cross-cutting — WebUI + CLI
-------------------------------
D10. Resolved: option B — preset picker + check enable/disable + per-
     check parameter editing via form fields in the check detail modal
     (Wave 6). Payload file management (option C) deferred.

D11. Config export / resolved view: Add a CLI command
     (chainsmith config resolve) and a WebUI panel that shows the
     fully merged config for a given preset? Useful for debugging
     "why did this check behave that way?"

Wave 6 — Check detail modals
--------------------------------
D12. How should checks declare their configurable params for the modal
     to render form fields? Options:
     (A) A CONFIGURABLE_PARAMS class attribute on each check — list of
         dicts with name, type, default, description, constraints
     (B) Infer from the YAML schema (Wave 1) — any key under the
         check's config section becomes a field
     (C) Both: checks declare params explicitly, YAML schema validates
     Option A gives better UI control (labels, help text, input types).
     Option B is less work but produces generic forms.

D13. Should per-check config changes made in the modal persist across
     page reloads (localStorage / API save), or are they ephemeral
     (lost on refresh, only submitted with the scan)? Persistent is
     more useful but needs a save/discard model. Ephemeral is simpler
     but frustrating if you configure 10 checks then accidentally
     refresh.

D14. Should the modal show the check's dependency graph? (i.e., which
     checks must run before this one based on conditions/produces.)
     Useful for understanding why a check is greyed out, but adds
     frontend complexity.

D15. Should there be a "Configure All" or bulk-edit mode for applying
     the same parameter change (e.g., timeout) across multiple checks
     at once? Or is that what presets (Wave 3) already solve?
